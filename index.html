Cross-Domain Assurance

The aim of this project was to provide a secure and efficient method of developing and testing cross-domain (Classified and Unclassified) communication and file transfers. This work was documented and implemented to enable future accredidation and verification of cross-domain solutions.

My work focused on the implementation of Apache Karaf and RabbitMQ.

Apache Karaf

Apache Karaf is an OSGi container that creates an Enterprise Service Bus for the bundles loaded into its environment. It allows for easier clustering, avoiding the creation of custom communication protocols. It has a bash environment and depends on Maven for loading its bundles (OSGi packages). In our environment, we load Camel and Zookeeper. The former is used for routing and the latter for clustering.

RabbitMQ

RabbitMQ is an Advanced Message Queuing Protocol (AMQP) application. It is used to guarantee delivery through message queue persistence across a cluster of nodes. Each instance of RabbitMQ maintains a list of messages waiting to be delivered, so if one node goes down, another can pick up where it left off.

Highly Available Messaging System

The aim of this project was to create a highly available messaging system. This meant that it had to be resilient to failure and that it would notify senders that the system had received each message. The system would then re-send the message until it recieveed a confirmation of delivery.

This system strictly could not fail. In order to accomplish this, myself and one other intern used the highly available development model to develop the enterprise service bus component of the system. Our core focus was the implementation of Apache Karaf and RabbitMQ.

Karaf

Karaf is an enterprise service bus (ESB). An ESB’s job is to serve as a translator for multiple programs that don’t necessarily expect the same data format.

RabbitMQ

RabbitMQ is an Advanced Message Queuing Protocol (AMQP) application. It is used to guarantee delivery through message queue persistence across a cluster of nodes. Each instance of RabbitMQ maintains a list of messages waiting to be delivered; if one node goes down, another can pick up where it left off.

Our Solution

A message is recieved by a clustered RabbitMQ node. This node sends confirmation of receipt to the sender and then redundantly stores the message across all RabbitMQ nodes. It then repeatedly sends the message to the Karaf cluster until a confirmation is received. This process is mirrored on the opposite side of the cluster.

Barring power or networking failure, this system has a near-zero chance of failure. Additionally, the Apache Karaf cluster allows for efficient message routing for custom and pre-built message translation and form-checking protocols.

MinerTrax

MinerTrax is the result of the frustration of myself and my roomate over missing the eBus to and from campus. To fix this, we built a web app that could show the location and route of the bus. The app even collects data that is used for research at our university.

My largest contribution to the project was the design and constrution of a device to keep track of the number of people using the bus. This was accomplished through the combination of an Arduino, two distance sensors, and a Bluetooth transmitter. The Arduino counts the people and whether they were boarding or exiting. It then transmits this information to a phone to be sent to the web server for analysis.
